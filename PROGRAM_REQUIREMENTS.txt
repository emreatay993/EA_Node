Performance Requirements:
Stack: PyQt6, QGraphicsScene/QGraphicsView, custom QGraphicsItem (no per-node QGraphicsProxyWidget for main graph).
Target scale: at least 1000 nodes and 5000 edges in one scene.
Responsiveness target: p95 frame time <= 33 ms during pan/zoom with target graph loaded.
Interaction target: drag/select/connect operations feel real-time with no multi-second stalls.
Startup target: open target graph in under 3 s on reference hardware.
Qt Rendering and View Settings

Use QGraphicsView::MinimalViewportUpdate (or benchmark against BoundingRectViewportUpdate).
Enable optimization flags: DontSavePainterState, DontAdjustForAntialiasing (if visual quality allows).
Use caching where beneficial: view background cache and item-level cache for expensive static visuals.
Fix scene rect explicitly; avoid frequent sceneRect recomputation.
Antialiasing policy: disable or reduce while panning/zooming; restore on interaction stop.
Benchmark item index mode: BspTreeIndex for mostly static scenes, NoIndex for highly dynamic scenes.
Node/Edge Item Requirements

paint() must be allocation-free in hot path (no per-frame object creation).
Precompute and cache edge paths (QPainterPath) and text layout.
Cache stable boundingRect() and shape() results.
Level-of-detail rendering: simplify nodes/edges/labels when zoomed out.
Only update dirty items; avoid full-scene invalidation.
Python Overhead Controls

Keep graph model separate from UI items.
Batch graph mutations; do not emit per-item update signals in large loops.
Coalesce frequent updates with short timer-based debounce (8-16 ms window).
Use lightweight Python objects (dataclass(slots=True) / __slots__) in core graph data.
Keep Python out of ultra-hot render loops; move hotspots to NumPy/Cython/C++ if profiling proves need.
No blocking work on UI thread.
Concurrency Rules

UI thread: rendering, input, scene mutations only.
Worker threads (QThreadPool/QRunnable) for compute-heavy tasks (layout, validation, analytics).
Cross-thread communication via queued signals only.
Profiling and Acceptance Tests

Must include scripted stress scenarios: load, pan, zoom, box-select, mass connect/disconnect.
Measure and report: FPS/frame time percentiles, mutation latency, memory usage.
Profiling tools: cProfile/py-spy plus in-app timers around load, layout, paint, and update phases.
Release gate: all performance targets met on reference hardware with target-scale graph.
