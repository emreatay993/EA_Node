# Ansys DPF Full Plugin Rollout

## Summary
- Yes, this is doable.
- The operator side is the strongest part: `ansys.dpf.core` operator specifications already expose pin names, accepted types, and `optional` status, so required vs optional port generation is realistic.
- The backend preparation is still necessary in six areas: versioned plugin lifecycle, multi-type port compatibility, source-metadata generalization for helper callables, DPF library taxonomy and classification, DPF runtime handle/materialization coverage, and cache/persistence portability.
- Chosen scope: **hybrid staged**. Build the backend so it can support a broader mirror later, but first ship:
  - all `dpf.operators.*` families
  - curated workflow helpers needed for end-to-end authoring: `DataSources`, `StreamsContainer`, `Model`, `Workflow`, `field_from_array`, `fields_factory`, `fields_container_factory`, `mesh_scoping_factory`, and `time_freq_scoping_factory`
- The library UI will be **workflow-first**, not a raw package dump. The first-wave taxonomy is:
  - `Ansys DPF > Inputs`
  - `Ansys DPF > Helpers`
  - `Ansys DPF > Operators`
  - `Ansys DPF > Workflow`
  - `Ansys DPF > Viewer`
  - `Ansys DPF > Advanced`

## Key Changes
- Add a **version-aware DPF plugin backend**. On every startup:
  - resolve `ansys.dpf.core.__version__`
  - compare it to the stored DPF plugin version
  - if different, rebuild the generated descriptor cache
  - persist the plugin version using the exact library version string
  - continue startup without prompting
- Keep plugin version state **app-global**, not project-global. Do not rewrite `.sfe` files just because the installed DPF library version changed.
- Extend the node contract for autogenerated DPF nodes:
  - support **multi-type input compatibility** instead of strict single `data_type` equality
  - support **callable-backed source metadata** for helper constructors/factories, not just operator-backed metadata
- Introduce a **DPF type normalizer**:
  - preserve specialized fast-path types for high-value objects (`model`, `mesh`, `scoping`, `field`, `fields_container`, `data_sources`, `streams_container`, `workflow`, `result_file`, `view_session`)
  - back the rest with a generic DPF object-handle path so broad operator coverage is not blocked on bespoke handle classes
- Add a **workflow-first DPF taxonomy** so the canvas library remains usable:
  - operators stay grouped by DPF family
  - helpers are grouped by workflow role
  - raw or noisy surfaces are reserved for `Advanced`
- Keep missing-plugin portability intact:
  - generated `dpf.*` nodes reopen as read-only placeholders when DPF is missing
  - saved labels, properties, exposed-port state, and edges remain preserved

## DPF Library Taxonomy
- **Inputs**
  - `Result Files`
  - `Raw Data`
- **Helpers**
  - `Models`
  - `Scoping`
  - `Factories`
  - `Containers`
  - `Support`
- **Operators**
  - family-based categories such as `Result`, `Math`, `Averaging`, `Invariant`, `Filter`, `Geo`, `Logic`, `Mapping`, `Compression`, and `Utility`
- **Workflow**
  - `Build`
  - `Execute`
- **Viewer**
  - viewer and visualization-specific nodes
- **Advanced**
  - `Server`
  - `Raw API Mirror`
- Classification rules:
  - operators are categorized by DPF operator family
  - helper and constructor nodes are categorized by workflow role rather than raw Python module name
  - original DPF module/operator paths remain in descriptor metadata so future UI reorganization does not break saved graphs
  - `Advanced > Raw API Mirror` is reserved for later literal-surface expansion and is out of scope for the first wave

## Public Interface Changes
- App preferences schema advances to `v3` and adds:
  - `plugin_backends.ansys_dpf_core.plugin_version`
  - `plugin_backends.ansys_dpf_core.catalog_schema_version`
  - `plugin_backends.ansys_dpf_core.cache_key`
  - `plugin_backends.ansys_dpf_core.last_indexed_utc`
- `PluginBackendDescriptor` gains optional backend-version resolution so version-aware generated backends do not special-case DPF in the loader.
- `PortSpec` gains `accepted_data_types: tuple[str, ...] = ()`; graph compatibility checks use this set when present.
- `NodeTypeSpec.source_metadata`, `PortSpec.source_metadata`, and `PropertySpec.source_metadata` are widened to support:
  - DPF operator metadata
  - callable-binding metadata for helper constructors/factories/mutators
  - family path and stability tags for library organization
- Stable generated node IDs:
  - operators: `dpf.op.<family>.<name>`
  - helpers: `dpf.helper.<module>.<callable>`
- Stable generated category paths:
  - `("Ansys DPF", "Inputs", "<family>")`
  - `("Ansys DPF", "Helpers", "<family>")`
  - `("Ansys DPF", "Operators", "<family>")`
  - `("Ansys DPF", "Workflow", "<family>")`
  - `("Ansys DPF", "Viewer")`
  - `("Ansys DPF", "Advanced", "<family>")`
- Generated descriptor metadata persists the original `ansys.dpf.core` source path, resolved family path, and stability level (`core`, `advanced`, `raw`).

## Execution Tasks

### T01
- **Goal:** Add version-aware DPF backend state and descriptor-cache invalidation.
- **Preconditions:** Existing optional DPF backend loads through the registry bootstrap path.
- **Conservative write scope:** app preferences, settings, plugin contracts/loader, DPF catalog bootstrap.
- **Deliverables:** startup version check, same-version fast path, changed-version rebuild path, persisted plugin version equal to `ansys.dpf.core.__version__`.
- **Verification:** tests for first load, unchanged version, changed version, corrupt/missing cache, and missing dependency.
- **Non-goals:** generating new operator/helper nodes.
- **Packetization notes:** `P01 Versioned Plugin Lifecycle`.

### T02
- **Goal:** Expand node/runtime contracts so autogenerated DPF pins and helper callables can be represented without hand-written one-offs.
- **Preconditions:** T01 complete.
- **Conservative write scope:** node spec contracts, registry validation, graph type-compatibility helpers.
- **Deliverables:** `accepted_data_types`, callable source metadata/binding specs, normalized DPF type-ID mapping, validation rules for optional/default/exclusive inputs.
- **Verification:** contract and validation tests for multi-type inputs, optional pins, property-backed defaults, and callable metadata.
- **Non-goals:** DPF-specific descriptor generation.
- **Packetization notes:** `P02 Contracts`.

### T03
- **Goal:** Introduce the DPF library taxonomy and split the catalog seams so operator and helper rollout can proceed on disjoint write scopes.
- **Preconditions:** T02 complete.
- **Conservative write scope:** DPF taxonomy constants, category mapping helpers, shared catalog wiring, and node-library regression anchors.
- **Deliverables:** top-level `Inputs`/`Helpers`/`Operators`/`Workflow`/`Viewer`/`Advanced` taxonomy, workflow-role helper categories, family/stability metadata tags, and split catalog entry points for operator and helper generation.
- **Verification:** taxonomy and registry-discoverability tests, plus updates to inherited DPF catalog category tests.
- **Non-goals:** generating the operator or helper descriptors themselves.
- **Packetization notes:** `P03 Family Taxonomy`.

### T04
- **Goal:** Generate the full operator catalog from `dpf.operators.*`.
- **Preconditions:** T03 complete.
- **Conservative write scope:** operator catalog/generator modules and operator-catalog regression anchors.
- **Deliverables:** family enumerator, stable operator IDs/categories, generation rules:
  - required inputs => exposed ports
  - optional object-like inputs => hidden optional ports
  - optional scalar/string/bool/enum inputs => properties with optional port exposure only when the type maps cleanly
  - outputs => exposed ports
  - mixed-type pins use `accepted_data_types`; pins that still cannot be ported cleanly fall back to property-only reflection
- **Verification:** generated spec snapshot tests, spot checks against live operators such as `result.stress`, and cache round-trip tests.
- **Non-goals:** helper module exposure.
- **Packetization notes:** `P04 Operator Families`.

### T05

## Test Plan
- Startup with DPF missing keeps the app usable and registers no generated DPF nodes.
- First startup with DPF installed builds the catalog cache and records plugin version `0.15.0`-style exactly.
- Startup with unchanged DPF version reuses cache and does not regenerate descriptors.
- Startup after simulated version change regenerates descriptors and updates stored plugin version only.
- The node library exposes the workflow-first taxonomy and keeps DPF operators under family-specific categories rather than a raw package mirror.
- `dpf.op.result.stress` generates required and optional pins correctly, with optional pins hidden by default.
- Multi-type pins accept only the declared `accepted_data_types`; invalid connections are rejected before execution.
- Curated helper chains execute on repo DPF fixtures.
- Save/reopen with DPF absent preserves generated nodes as read-only placeholders with edges, properties, and exposed-port state intact.

## Assumptions
- "Expose all subpackages" means **full operator coverage plus curated workflow helpers first**, not a literal first-wave mirror of all ~900 public importable modules.
- Plugin version state lives in app preferences, not project metadata and not package manifests.
- The plugin version string is always the exact installed `ansys.dpf.core.__version__`.
- Missing DPF does not erase cached metadata or mutate stored project content; it only makes the backend unavailable.
- Generic instance-method reflection is deferred; only explicitly curated helper mutators ship in this wave.
