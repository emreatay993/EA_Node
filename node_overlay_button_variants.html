Node Overlay Button — UX Variants

Nine ideas for replacing always-visible inline buttons with hover-revealed overlays. Hover each node to trigger the animation. Each variant trades discoverability, screen real estate, and interaction precision differently — pick the mix that fits how your users scan a dense graph.

Cross-cutting considerations

  • Selection sticky-state. When a node is selected, keep the overlay visible even without hover — users expect the selected node to stay "active".
  • Keyboard parity. Whichever variant you pick, ensure Tab/Enter reaches overlay actions when the node is focused; hover-only kills a11y.
  • Hover grace period. Add a ~120ms delay before hiding so users can travel from node → overlay without losing it (classic "menu intent" pattern).
  • Edge flipping. V1, V3, V9 need smart flip when near viewport edges, or they'll overflow.
  • Zoom behavior. Decide if overlay buttons stay screen-sized (constant readable) or scale with the node — mixing them (icons scale, but min-size clamped) usually feels best.
  • Running / Halted states. The OPEN and HALTED badges convey state, not action. Keep those always-visible even if action buttons hide on hover.
My picks for this editor: V2 (corner stack) as the primary pattern — safest, highest fit with your current chrome and tightest against the existing GraphSurfaceButton component. V1 (top toolbar) if you want a more expressive "this node is focused" moment. V5 (inline title reveal) if you want zero overlay plumbing and only need 2–3 actions.
