# DPF_OPERATOR_PLUGIN_BACKEND_REFACTOR P01: Optional DPF Plugin Lifecycle

## Objective

- Make the DPF node family optional, lazy, and availability-aware so the app starts cleanly without `ansys-dpf-core` while preserving a stable plugin contract for later autogenerated operator rollout.

## Preconditions

- `P00` is marked `PASS` in [DPF_OPERATOR_PLUGIN_BACKEND_REFACTOR_STATUS.md](./DPF_OPERATOR_PLUGIN_BACKEND_REFACTOR_STATUS.md).
- No later `DPF_OPERATOR_PLUGIN_BACKEND_REFACTOR` packet is in progress.

## Required Behavior

- Introduce a plugin availability contract that can represent an installed DPF backend and a missing-dependency DPF backend without crashing startup.
- Keep DPF registration lazy so the loader can defer DPF-specific work until availability is known.
- Preserve current non-DPF node/plugin boot behavior.
- Preserve current handwritten DPF node families behind the new availability gate; do not introduce autogenerated operator expansion yet.
- Update inherited plugin-loader and DPF catalog regression anchors in place when the plugin contract changes.

## Non-Goals

- No DPF operator metadata normalization yet; that belongs to `P02`.
- No compute/runtime bridge rewrite yet; that belongs to `P03`.
- No missing-plugin placeholder rendering or persistence adoption yet; that belongs to `P04`.

## Acceptance Criteria

- Startup and plugin loading no longer require `ansys-dpf-core` to be installed.
- The plugin contract can report DPF unavailable state without masking the rest of the node catalog.
- Existing handwritten DPF node families remain gated behind the new availability contract.
- The inherited loader and DPF catalog regression anchors pass.
