# DPF Operator Plugin Backend Refactor QA Matrix

- Updated: `2026-04-12`
- Packet set: retained `DPF_OPERATOR_PLUGIN_BACKEND_REFACTOR` (`P01` through `P05`)
- Scope: final closeout matrix for the optional DPF plugin lifecycle, normalized operator metadata contract, generic operator-backed runtime adapter, missing-plugin placeholder portability, and retained docs or traceability evidence.

## Locked Scope

- `ansys-dpf-core` remains optional. Startup without DPF must keep the rest of the app usable.
- This wave is backend preparation only. The shipped DPF scope remains the existing handwritten compute nodes plus `dpf.viewer`; no broad autogenerated operator exposure lands here.
- The authoritative backend contract now spans lazy plugin availability, normalized operator and port descriptors, generic operator-backed runtime invocation, and read-only missing-plugin placeholders for reopened projects.
- The generated-port semantics must distinguish required inputs, optional inputs, omitted operator defaults, explicit literal values, and mutually exclusive input groups.
- Missing-plugin placeholders preserve saved labels, values, exposure metadata, and edges, but they stay non-executable and cannot expose live schema editing until DPF is available again.
- Repo-local `.rst` and `.rth` fixtures under `tests/ansys_dpf_core/example_outputs/` remain the authoritative smoke inputs for packet-owned DPF verification.
- Broad autogenerated operator rollout and non-operator `ansys.dpf.core` reflection remain deferred to a later packet set that reuses this contract rather than replacing it.
- The handwritten `dpf.field_ops` `ElementalNodal` passthrough remains intentionally unchanged in this wave.

## Residual Risks

- `P05` is docs-and-proof only. It does not rerun a new broad DPF aggregate outside the retained packet verification commands and the declared closeout gates.
- The packet set intentionally freezes the preparation contract rather than shipping mass autogenerated operator exposure or non-operator `ansys.dpf.core` reflection.
- The existing handwritten `dpf.field_ops` `ElementalNodal` passthrough remains in place because this wave generalizes the operator-backed runtime seam without rewriting every handwritten DPF fallback.
- Unknown historic `dpf.*` nodes that lack enough saved metadata still fall back to a minimal placeholder until a live plugin can supply the authoritative schema.
