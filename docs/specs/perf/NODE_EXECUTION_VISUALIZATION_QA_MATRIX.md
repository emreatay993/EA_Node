## Locked Scope

- The authoritative visual reference is `docs/node_execution_visualization_alternatives.html`, and the shipped manual baseline remains the A+E-hybrid visual baseline paired with authored control-edge dim-before-progress and first-progress flash behavior on the graph canvas.
- Running/completed node chrome remains protocol-neutral: `node_started` drives the blue pulse halo and live elapsed timer, `node_completed` drives the green completed flash/border, authored `exec` / `completed` control-edge progress, and cached elapsed overwrites for the completed node, and `node_failed_handled` progresses authored `failed` control edges when execution continues through handlers.
- Worker timing metadata is additive and backward compatible: `node_started` may include `started_at_epoch_ms`, `node_completed` may include `elapsed_ms`, and the shell fallback path computes elapsed time from its own transient started-at state when those fields are absent.
- Running/completed node chrome, authored control-edge progress, transient started-at state, and cached elapsed values stay active-workspace-filtered and in-memory only. The feature does not add `.sfe` persistence fields, session-restore payloads, project-file schema changes, or a second execution-visualization signal/revision channel.
- The per-run authored-edge lookup is frozen from the run-start authored workspace snapshot so the active workspace canvas stays aligned with the run snapshot even if the graph is edited mid-run.
- Running nodes use a blue pulse halo. Completed nodes emit a green completed flash and then keep a static green border until the run ends. During an active run, authored control edges that have not progressed render at `0.2` alpha and `1.4px`; the first progress transition returns the edge to its normal interaction-aware styling and emits a QML-local `240ms` base-color flash using a `+2.4px` overlay.
- Selection and preview keep their existing interaction colors and widths and must not be dimmed, but the one-shot flash can still layer on top. Data edges and passive `flow` edges remain unchanged.
- Cached elapsed values are session-only and per-workspace. `run_started` clears transient running/completed visualization and transient started-at state for the new run, but it preserves cached elapsed values for an otherwise unchanged workspace.
- `node_completed` overwrites the cached elapsed value for that node only. Stopped or failed runs update only the nodes that actually reached completion; untouched nodes keep their prior cached elapsed value until execution-affecting invalidation or session replacement clears it.
- Execution-affecting history commits, undo, and redo invalidate cached elapsed values. Cosmetic/layout edits such as move, resize, collapse, rename/title-only changes, style edits, edge-label edits, port-label edits, comment-only changes, selection changes, scope changes, workspace switching, or theme/performance changes preserve the cache.
- Completed nodes may keep a quieter footer that renders cached elapsed text after transient running/completed chrome clears, but that persistent cached footer remains shell/QML-owned, active-workspace-filtered, and outside `.sfe` persistence.
- `run_completed`, `run_stopped`, fatal worker-reset failures, and a fresh project session clear node chrome, edge progress, and transient started-at state together. Non-fatal `run_failed` preserves the last execution context, while handled failures keep the current run visualization until the run actually ends. Project replacement, new-project install, and equivalent session replacement flows clear all cached elapsed timing data.

## Remaining Manual A+E-Hybrid, Execution-Edge, and Persistent Elapsed Checks

1. Live timer and cached footer: open a desktop Qt session on a workspace with a simple runnable chain, start a run, and confirm the active node shows the blue pulse halo plus a live elapsed timer; after completion, confirm the node keeps the quieter cached footer text after transient run chrome clears.
2. Edge progress and retained completed border: let a straightforward successful path run and confirm each newly progressed authored control edge returns to normal styling and emits one short base-color flash (`240ms`), while each completed node emits a green completed flash once and then keeps a static green border until the run ends.
3. Handled failure and partial cache overwrite: route execution through a handled `On Failure` branch or stop a run mid-workflow and confirm only nodes that actually completed keep cached elapsed values, while currently running or failed nodes do not retain stale footer text.
4. Execution-affecting invalidation versus cosmetic edits: move, rename, or restyle a completed node and confirm the cached footer remains visible; then change an execution-affecting node property or structural execution edge and confirm the relevant cached elapsed values clear immediately.
5. Workspace and session isolation: switch to a different workspace while a run is active or immediately after completion and confirm foreign node/edge highlights and cached elapsed footers do not appear on the inactive workspace; open a different project or install a new project session and confirm all cached elapsed timing data clears.
6. No persistence after save/reopen: save the project, reopen the same `.sfe`, and confirm prior cached elapsed values do not persist across the fresh session even though the retained QA matrix and traceability docs now record the feature.

## Residual Desktop-Only Validation

- Offscreen automated coverage does not validate final pulse/flash intensity, perceived footer legibility, or border/alpha contrast across real Windows desktop compositing and dense-graph zoom levels.
- Shell fallback timing parity when worker timing metadata is absent is covered by protocol and controller regressions; manual desktop acceptance still needs to confirm there is no visible discontinuity on any non-standard worker surface family.
- The live timer, cached footer, and `240ms` edge flash are intentionally shell/QML-local and session-only, so they reset if the relevant host visual is destroyed or the project session is replaced; later persistence work should keep that no-`.sfe` constraint explicit rather than piggybacking on execution chrome state.
